KG context is a real edge — but the lever is the subagent prompt, not the KG itself.

Populating the KG adds modest F1 lift (+0.04). Editing one paragraph in the code-review subagent prompt to push graph-first verification adds another +0.09 — triple the impact of having the data. Wrong wording on the same lever destroys recall.

1

The question

Does giving the production code-review agent access to a Knowledge Graph (callers, impact, co_changes, prior findings, linked issues) produce a measurably better code review than the prod skill alone? If so, by how much, and via what mechanism?

Five sequential phases, each isolating one variable. Same 10 PRs, same model (zai-coding-plan/glm-5.1), same scoring (F1 vs answer-key extracted from prior bot review verdicts). Local backend MCP, local Neo4j, local Firestore emulator throughout.

2

Five phases, one lever found

Each phase changed exactly one variable from the previous. The lever — the subagent prompt template inside code-review/SKILL.md — wasn't found until Phase D.

A

Production skill, empty KG slice

orig F1 0.233 · ext F1 0.260

Run prod code-review + explore-context against local MCP. Local KG had only PR/Finding data for cr-loop — no code-domain entities yet.

Observation: the agent organically called aictrl_query_context 55 times — but every call returned count: 0. The agent reaches for the graph even when nothing answers.
55 KG calls 0 non-empty results 37 findings
B

Production skill, KG populated

orig F1 0.304 · ext F1 0.300

Indexed aictrl repo into local Neo4j with org_id='cr-loop' (focused index of the 10 PRs' touched files + siblings). Two real bugs in the indexer fixed in the process: .claude not in EXCLUDED_DIRS; push(...spread) blew the call stack on big edge sets.

Lift: +30% F1 vs Phase A on the original answer-key — concentrated entirely in PRs that scored zero in Phase A (1783: 0→0.44, 1782: 0→0.25, 1794: 0.48→0.69).
63 KG calls 2,060 non-empty results 41 findings 0 graph-traversal calls
C

Rewrite explore-context.md — killed early

no propagation

Wrote v2.0 of explore-context.md with a "Required Pre-Recording Verification Protocol", reordered actions to put callers/impact/co_changes first. After 5 PRs the agent's tool selection had not changed.

Diagnosis: subagents are spawned with a self-contained prompt (PR diff + checklist) and do not auto-load the orchestrator's -f skills. Editing explore-context.md alone has zero effect on Bug Hunter behaviour.
behaviour unchanged terminated 5/10
D

Edit subagent prompt: "MUST verify before recording"

orig F1 0.143 · ext F1 0.149

Found the smoking gun — the orchestrator's "Tool rules" block told subagents "You MAY call query_context to read full-file context, keep under 5 calls." Replaced with imperative graph-first guidance + "if 0 callers, drop the finding".

Lever found, wrong wording: 165 graph-traversal calls (0 → 165) but recall collapsed. Findings dropped from 41 → 15. Precision climbed to 1.0 across all PRs but the agent gated emission on verification.
165 graph-traversal calls 15 findings (was 41) F1 −47% vs B
E

Soften the rule: "SHOULD use, no gating"

orig F1 0.243 · ext F1 0.388 ⭐

Reposition graph actions as severity / scope / suppression informers, not pre-recording gates. Removed "drop if 0 callers" — replaced with "0 callers is a candidate for dead-code annotation but the original finding still stands if valid".

Best result: 70 graph-traversal calls (kept the lever's intent) AND 43 findings emitted (highest of any phase). Original-AK F1 looked like a regression because the agent caught novels — until the methodology correction in section 3.
70 graph-traversal calls 43 findings (highest) +29% F1 vs B (extended AK)
3

The methodology twist — the answer-key was wrong

The original answer-key was built from prior bot reviews — it captures what bots have flagged before. When agents reach for graph context, they catch things bots didn't. Those novels score as 0 against the bot-AK. Triaging the 62 deduplicated novels from triage changed the picture materially.

Original answer-key

83
labelled findings · built from bot reviews
2 of 10 PRs had 0 AK entries (no human review)

Extended answer-key (after triage)

143 +60
labelled findings · 60 novels manually verified TRUE
57 TRUE / DEFER (consistency, refactor) 3 TRUE / FIX (real bugs bots missed) 2 dropped FALSE
Methodology bias: any production trend dashboard comparing KG-on / KG-off against a bot-built answer-key will systematically under-credit KG-augmented agents. Three real bugs (formatDuration multi-day truncation, hasFindingsMin enum mismatch, skills.ts missing trim) emerged from the novel triage — none flagged by bots originally.

Phase B is the production-equivalent: prod code-review skill verbatim, KG populated, the original "MAY call query_context to read full-file context (under 5 calls)" rule. Phase E swaps that one paragraph. The shift in query_context action distribution is the mechanism behind the F1 lift.

Per-action breakdown. The four bold rows are the graph-traversal actions that the rewritten prompt explicitly asked for. They went from 0 calls to 69 — every one of them previously unreachable through the agent's organic behaviour.

All tools across the sweep. Phase E doesn't just use KG more — it does more filesystem work too (bash, read, grep). The KG and the filesystem are complementary: graph queries find candidates, file reads verify them. The cost story is "additive context", not "compute substitution".

The mechanism in one line: the new prompt unlocks four graph-traversal actions that the prod skill never reached for organically (callers, impact, co_changes, issues), and triples total KG usage. Those new query types — not raw call volume — are what move the F1 needle.

Token & latency cost

The +0.09 F1 lift is not free. Phase E uses ~1.9× the tokens and ~1.8× the model steps of Phase B. Output tokens (the agent's "work") only grow ~1.1× — the bulk of the extra cost is input/cache reads from larger context windows as KG responses accumulate.

Per 10-PR sweep Phase B (current prod-equiv) Phase E (proposed) Δ
Total tokens 24.1M 46.4M +1.93×
Input tokens (fresh, non-cached) 2.10M 3.32M +1.58×
Output tokens (agent work) 537K 593K +1.10×
Cache reads (cheap) 21.5M 42.5M +1.97×
Model steps (~latency proxy) 712 1,268 +1.78×
Per-PR avg tokens 2.41M 4.64M +1.93×
The trade-off: ~1.9× cost for +30% F1 (extended AK) and roughly +50% TPs caught. On the GLM coding plan this is free; on Anthropic / OpenAI usage-pricing the ratio matters. Cache hits (which dominate the token mix) are typically priced ~10× cheaper than fresh input — actual $/review ratio is closer to 1.7× than the raw 1.93×.
6

Three lessons

1

The lever is in code-review/SKILL.md, not explore-context.md

Subagents do not inherit the orchestrator's -f loaded skills. They get a self-contained prompt + the MCP tools. Phase C proved explore-context.md edits don't propagate. Phase D proved subagent-prompt edits do.

Production implication: the prod code-review skill is the actual control surface for KG behaviour. Keep explore-context.md as agent-side reference; don't expect it to shape subagent decisions.

2

KG availability is necessary; the wording around it is sufficient

+0.04 F1 from KG availability (B vs A). +0.09 F1 from the right prompt wording (E vs B). −0.15 F1 from the wrong wording on the same lever (D vs B). Total swing controlled by one paragraph: 2.6×.

The dangerous failure mode is "MUST verify before recording" — it gates emission and collapses recall. The safe framing is "SHOULD use to inform severity / scope / suppression — but record findings when you see them".

3

Bot-built answer-keys systematically under-credit KG-augmented review

When agents reach for blast radius, cross-PR history, and historical coupling, they catch findings bots typically don't flag. Against an AK built from bot reviews, those novels score as 0. Phase E was the winner all along; the original AK hid it.

The methodology gap reduced Phase E's measured F1 by 0.145. Until production telemetry includes a novel-triage path, every iteration that adds context will appear to regress.

7

What to ship

Three concrete actions, ordered by leverage. Each is a small diff today, not a roadmap-scale build.

1

Replace the subagent "Tool rules" block in data/default-skills/code-review/SKILL.md

Swap the current "MAY call query_context to read full-file context, keep under 5 calls" for the Phase E "SHOULD use to inform findings" wording. One-paragraph diff. Worth +0.088 F1 on this 10-PR sample.

2

Verify production executor's KG slice has code-domain coverage

The Phase A → Phase B delta (+0.04 F1) only materialises when query_context: domain=code returns real results. Confirm the prod execution path indexes its org's repository — the focused indexer pattern from this experiment (scripts/index-pr-files.ts) is the working blueprint.

3

Build a novel-triage feedback loop into the production trend dashboard

Surface emitted-findings-without-AK-match. Flag them for human triage. Feed confirmed-true ones back into the AK. Without this, every future iteration that adds context will look like a regression — exactly what happened in this experiment until we re-triaged manually.

What to defer: new MCP domain types, additional graph actions, new tool surfaces (#1810 et al.). The dominant driver here was the subagent prompt phrasing — much cheaper than tool-surface changes. Re-evaluate after iterating once on the prompt with 3-seed averaging.
